Each topic begins with a short introduction or abstract that describes the contents of the topics. This topic is an example topic that shows all styles used in the user manual and explains their usage. It also gives general information on how to write documentation.

Setting your H&M environment

Page width

Since we plan to publish this manual as PDF also the page width is an issue. To have a good guess on how the documentation will look when published as PDF it is suggested to size the H&M text editor area so that it shows about 18 cm of width (ruler units can be set in user settings) Pöyry

Spell checking

We use English (US) as documentation language therefore you should set the spell checker accordingly. For <%PRODUCTLINE%> specific words we use an additional dictionary G:\06 - Fusion\Documentation\Fusion additional dictionary.adu all new entries should go to this dictionary.

Topic ID guidelines

The topic id is important since it will be part of the URL when published to HTML.

•Once published Topic IDs shall not change since we might have links pointing to the topic.

•Random Ids (like GUIDs) are not used since finding a topic internally in H+M (e.g. when linking) is then a nightmare

•Topics ID shall not express the chapter or hierarchy in which the topic is since that might change and then a rename would be necessary

•Do not use CamelCasing: when publishing to html the file names will be all lowercase, therefore camelcasing is gone.

•Word-Separation: if topic consists of more tha one word use the minus sign to separate.

•Prefixes: Sometimes we want to prefix or group topics in order to prevent duplicates. e.g. all topics for smart client windows start with SCW- use an acronym then a minus

Creating Topics

When creating a new topic be sure to select the Default.template.xml as Topic template. This way the first Text you write is already in "Paragraph Mode" (see image below). Each topic has metadata (called Topic Eigenschaften, for us following meta data fields are important:

•Topic name (ID): This is the tricky part. Every topic needs a unique identifier and this identifier should be readable since we refer to it in linking of topics.
The easy guess would be to use the full path of the topic like "Chapter1_Chapter11_TopicA" but this approach has a downside: When we restructure things we have to change all topics IDs too.

•Keywords: These are the keywords used to make up an index for the help file and also for pdf files etc. We should fill these keywords but use them wisely.

We do not use A-keywords

Screenshots & images

Generally (at least in the product reference) we try to not add too many images. Where an image is worth a thousand words we will surely use an image, but remember that keeping images up to date is really painful. Images need to be stored in the Help&Manual project folder under the sub folder Baggage. Images must be stored in png format.

Image size is crucial since the help might be displayed online and we also do not want to get a huge pdf file. H&M has an integrated image editor that can be used to crop and resize an image.Do not simply resize the image in the topic but resample the image in the image editor! We avoid to add any arrows or other graphical and textual information to the image itself because this is extremely hard to update!

It is important to be able to identify to which topic images belong (on the files system) therefore following naming rules should be used for images: <TopicId>_<ImageNoOrName>.png.

Help&Manual automatically assigns the Image Caption style to all images. Image caption (description of images) go below the image.

Headers within topics

Within topics we use TopicHeading1 and TopicHeading2 to structure the text. Although we define TopicHeading2 it is usually a hint to create a better structured table of content when you need to use TopicHeading2.

Identifier

An identifier is used to mark a specific part of a text as being an identifier in the current context and not being part of the text itself. You use identifiers to refer to Screen elements etc. E.g. select Overwrite to overwrite existing observations.

Anchors within topics

Linking of other topics

Where ever possible we link other topics to give the user the possibility to easily navigate between topics. Following rules should be followed when linking:

•Give each link a meaningful name, do not use see here etc

•One destination should only be linked once in a topic

•If possible link to a specific anchor in the destination.

Usually only the first occurance of a keyword (e.g. feature) shall be linked

Lists

Unfortunately H&M does not support bullet lists or numbering lists as part of  styles. Therefore a there is a separate style named List that we use for all lists and then we apply the numbering or bullet style via the lists menu. We agree to use only simple bullets and numbers.

Example for bullet list:

•Lists shall never follow directly after a TopicHeading.

•List items shall start with an uppercase character

•Item 3

Example for numbering list:

1.Item 1

2.Item 2

3.Item 3

We do not use nested lists.

Notes, Warnings, Important Information

There might be the need to specially layout important notes or warnings and other important information. For this reason we define two styles for

Note: We use notes to underline the importance of a specific information. Notes always begin with the text Note:

Warning: We use warnings when the user should be aware that some action is risky in terms of information security (i.e. data loss, authorization authentication etc). Warnings always begin with the text Warning:

Permission information

Permissions: Throughout the documentation we will repeatedly document the permissions needed for some action. These permissions stuff must be formatted with the  specific style permissions. Permission documentation always begins with the text Permissions:

Tables

Tables can be helpful in many cases. Help&Manual supports table styles. Every table must use one of the predefined styles. Unfortunately these styles are rather limited therefore we additionally define styles for the contents of the table.

Other notes

Structuring the manual

For structuring we use folders with text. The text for a folder should at least contain a short summary of the folder's contents

Capitalization

Topic headers are not capitalized. a valid topic header would be "This is my new topic header" Although names etc are still capitalized E.g. "This is an introduction to the Application Server"

See also section

The see also section is a special section at the end of each topic that allows to link to other topics. Generally we try to link topics from with in the text but still there might be topics you want to point the user to. For this section we have a special Header format called See also. The header text shall always be See also.
